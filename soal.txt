Backend Developer Technical Test
Internal Procurement System

Studi kasus
Perusahaan X ingin membangun Sistem Procurement Internal untuk menggantikan
proses manual berbasis email.
Proses Bisnis Saat Ini:
1. Karyawan mengajukan permintaan barang/jasa
2. Purchasing melakukan verifikasi
3. Atasan purchasing melakukan approval
4. Warehouse melakukan pengecekan ketersediaan stok
5. Jika stok tidak tersedia → Purchasing melakukan pengadaan ke vendor
6. Status proses diinformasikan ke pemohon hingga selesai

Technical Stack
● Laravel 12 (API-only)
● PostgreSQL 14
● Token-based Authentication

REQUIREMENT ANALYSIS (20%)
1. Identifikasi:
● Actors
● Core Entities
● Workflow States
● Critical Business Rules
2. Tuliskan:
● Pertanyaan klarifikasi yang perlu Anda ajukan sebelum implementasi
● Asumsi yang Anda buat jika requirement belum lengkap

SYSTEM DESIGN (30%)
1. Buat ERD dan Struktur Tabel PostgreSQL
Minimal mencakup:
● users
● departments
● requests
● request_items
● approvals
● stock
● vendors
● procurement_orders
● status_history
Sertakan:
● Primary Key dan Foreign Key
● Strategi indexing
● Alasan penggunaan soft delete atau hard delete

2. Jelaskan:
● Alasan desain struktur tersebut
● Cara mencegah data redundancy
● Cara menjaga data consistency

WORKFLOW & BUSINESS LOGIC (20%)
1. Buat State Transition Diagram
Contoh state:
DRAFT → SUBMITTED → APPROVED → REJECTED → IN_PROCUREMENT →
COMPLETED
2. Jelaskan:
● Bagaimana mencegah double approval
● Bagaimana menangani parallel modification
● Bagaimana mencegah stock race condition
● Jika dua user membuka request yang sama bersamaan, bagaimana menghindari
konflik?

API DESIGN (15%)
Desain endpoint berikut:
● POST /requests
● GET /requests?status=approved
● POST /requests/{id}/approve
● POST /requests/{id}/reject
● POST /requests/{id}/procure
Untuk setiap endpoint jelaskan:
● Request body
● Response body

● HTTP status code
● Validasi utama

REPORTING & QUERY (10%)
Buat Query PostgreSQL untuk:
1. Top 5 department dengan permintaan terbanyak dalam 3 bulan terakhir
2. Kategori barang paling banyak diminta per bulan
3. Average lead time dari SUBMITTED hingga COMPLETED

ADVANCED ENGINEERING (5%)
Jawab secara singkat:
1. Jika sistem memiliki 5 juta record request, strategi optimasi apa yang akan Anda
gunakan?
2. Kapan perlu menggunakan partitioning?
3. Apakah perlu caching? Jika ya, bagian mana dan mengapa?

Pertanyaan tambahan (Nilai tambah)
● Bagaimana membuat audit trail yang immutable?
● Bagaimana desain sistem ini jika dijadikan microservice?
● Bagaimana menangani event asynchronous seperti notifikasi email?
